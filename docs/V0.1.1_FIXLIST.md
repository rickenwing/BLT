# v0.1.1 fix list

Running list of issues found during the v0.1.0 three-machine manual test
(`docs/TESTING_CHECKLIST.md`). Fix these, then cut v0.1.1. Append as testing
continues.

## Confirmed bugs

### BUG-1 — Admin "Settings" save always 422s; server paths can't be set via the UI

- **Symptom:** Saving library/staging/shared-pool paths (or any Settings
  field) returns HTTP 422; nothing appears in the server log.
- **Reported guess (wrong):** spaces/special characters in the paths. Not the
  cause — JSON strings carry those fine. The failure is independent of path
  content; *every* save 422s.
- **Root cause:** `GET /api/config` emits `jukebox_order_mode`
  (`crates/server/src/admin_api.rs:284`). The Settings page loads the whole
  object and PUTs it back verbatim — `api.put("/api/config", cfg)`
  (`admin-web/src/pages/Settings.tsx:93`). But the server's `ConfigPatch`
  struct uses `#[serde(deny_unknown_fields)]`
  (`crates/server/src/admin_api.rs:290`) and has no `jukebox_order_mode`
  field. Axum's `Json` extractor rejects the body with 422 **before**
  `put_config` runs, so the handler's logging never fires (explains the
  silent log). `jukebox_order_mode` is the only field GET emits that PUT
  rejects; it's owned separately by the Jukebox page via `/api/jukebox/mode`.
- **Recommended fix:** make GET/PUT symmetric. Add
  `jukebox_order_mode: Option<String>` to `ConfigPatch` and apply it (it's a
  real `Config` field), so a full-object round-trip is valid. Alternative:
  drop `jukebox_order_mode` from the GET response (Settings doesn't edit it).
  Add a regression test that round-trips GET→PUT unchanged.
- **Workaround during testing:** set the paths in
  `<data_root>/server/config.toml` (`library_path`, `staging_path`,
  `share_path`) and restart; config.toml is the authoritative store.
- **Severity:** high — server is unconfigurable through the UI without it.

### BUG-2 — Duplicate server entries in client discovery (manual + discovered for the same box)

- **Symptom:** The client's connect list shows the same server twice — one row
  badged `discovered`, one `manual`, identical game/share endpoints
  (screenshot: 192.168.50.17:7400/7401 listed twice).
- **Root cause:** Manual connect and mDNS discovery both write the same
  `servers` table, deduped only on `uuid`
  (`apps/desktop/src-tauri/src/db.rs:32-39`, `upsert_server` 93-116). A manual
  entry has no uuid → inserts a `uuid=NULL` row via a bare `INSERT` (no
  `ON CONFLICT`). mDNS later upserts a row keyed on the real uuid. Same box,
  two rows. The badge is purely `s.uuid ? discovered : manual`
  (`apps/desktop/src/client/Settings.tsx:188`). Secondary: repeated manual
  connects to the same IP pile up more NULL-uuid rows (no endpoint dedup).
- **Recommended fix:** dedup on `game_endpoint`. On discovery, if a uuid-less
  row with the same `game_endpoint` exists, adopt it (set its uuid/label)
  instead of inserting a new row; and make manual upsert dedup on
  `game_endpoint` so reconnecting doesn't duplicate. Optionally collapse rows
  sharing an endpoint in `list_servers`, preferring the discovered one.
- **Severity:** low/cosmetic — both Connect buttons reach the same server.

### DOC-1 — `info.json` schema was unclear; tester guessed wrong keys *(addressed)*

- **Issue:** No shipped example, so the test sidecar used `label` (should be
  `name`), `version` (no such field — auto-assigned), and `cover_b64` (cover is
  a separate `cover.*` file, not inline). Game still published; title fell back
  to folder name because `name` was null.
- **Done:** added `examples/README.md` (annotated schema + common mistakes) and
  `examples/sample-title/.blt/info.json` (copy-paste template), and the release
  workflow now ships `examples/` inside the server tarball. Lands in v0.1.1.

## Observations / to-verify

### OBS-1 — mtime-only change bumps manifest_ver and forces full re-download

- `scan_title` latches `any_change = true` whenever a file's size **or mtime**
  differs from the previous manifest (`library.rs:217-228`), *before*
  re-hashing. So a `touch` (identical content, new mtime) bumps the version,
  rebroadcasts `TitleChanged`, and makes clients re-download unchanged bytes.
- Not a correctness bug (versions stay immutable; integrity holds) and
  HARD CONSTRAINT #10 is intact (verified: sidecar-only edits go through
  `update_info_only`, no bump, tested by `rescan_bumps_version_sidecar_change_does_not`).
- Wasteful at 500 GB scale. Possible refinement: after re-hashing the changed
  set, compare the resulting chunk hashes to the previous manifest and skip the
  version bump if content is byte-identical (mtime-only). Low priority for
  hobby scale; note before relying on `touch`-safe rescans.
